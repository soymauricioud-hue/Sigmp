I. NATURALEZA DEL ÁRBOL DOCUMENTAL

El árbol documental del SMGIP no es una lista de documentos.

Es una arquitectura de política pública diseñada para construir, validar, versionar e implementar un sistema municipal complejo sin improvisación, sin captura institucional y sin activación prematura de capacidades regulatorias.

El árbol cumple seis funciones simultáneas:

Función de ingeniería inversaPermite descomponer el documento maestro SMGIP_version_final_completa en documentos fuente, documentos derivados, instrumentos operativos y salidas por fase.

Función de precedenciaDefine qué debe escribirse antes y qué no puede escribirse todavía. Las dependencias no son referencias decorativas: son reglas de desbloqueo documental.

Función de control de faseDefine qué documentación mínima permite declarar Fase I, qué documentación activa Fase II y qué documentación permite declarar Fase III.

Función anticapturaEvita que una persona, área, asociación, ONG, MVZ, criador, proveedor o autoridad concentre diseño, operación, validación, financiamiento, inspección y auditoría del mismo proceso.

Función de trazabilidadObliga a que cada acto clínico, administrativo, financiero, territorial, sancionatorio o de egreso pueda reconstruirse desde documento, responsable, evidencia y versión.

Función de uso controlado de IAPermite usar IA como herramienta de fragmentación, clasificación, auditoría, redacción asistida y verificación, sin que la IA invente estructura, active documentos no desbloqueados o confunda salidas de presentación con documentos fuente.

II. REGLA MATRIZ

El árbol debe leerse con esta regla:

Si un documento define una regla, se escribe antes.Si un documento aplica una regla, se escribe después.Si un documento mide una regla, se escribe después de la operación mínima.Si un documento sanciona una regla, se escribe solo cuando ya existen la regla, el dato, el procedimiento, la autoridad y la evidencia.

III. DOCUMENTO FUENTE CANÓNICO

El documento fuente canónico es:

SMGIP_version_final_completa

De este documento se extraen, por ingeniería inversa, los documentos 0.1, 0.2, 0.3, etc.

El repositorio Git no inicia como un espacio vacío. Inicia como un repositorio de fragmentación y trazabilidad del documento fuente canónico.

La primera tarea no es redactar documentos nuevos.La primera tarea es mapear el documento maestro hacia el árbol documental.

IV. DEFINICIONES OBLIGATORIAS DEL ÁRBOL

Documento maestroEs el documento fuente canónico. Contiene la visión integrada del sistema. No se trata como borrador suelto ni como texto inspiracional. Es la fuente de verdad inicial.

Documento fuenteDocumento que define reglas, conceptos, criterios, fases, estructura, autoridad o metodología. Otros documentos dependen de él.

Documento derivadoDocumento que toma reglas de documentos fuente y las adapta para un uso específico: operación, auditoría, campo, cabildo, ONG, ciudadanía, capacitación o convenio.

InstrumentoFormato, acta, checklist, bitácora, constancia, licencia, ficha o plantilla. No define política pública; ejecuta una regla ya definida.

DependenciaCondición que determina si un documento puede escribirse. Si la dependencia no existe o no está estable, el documento dependiente solo puede tener ficha, índice y criterios de desbloqueo.

Dependencia duraDocumento sin el cual otro no puede escribirse de forma sustantiva.

Dependencia jurídicaDocumento que permite validar obligaciones, sanciones, licencias, inspecciones o facultades.

Dependencia operativaDocumento que permite ejecutar un proceso en campo, clínica, UCR, campaña, CNVR o inspección.

Dependencia analíticaDocumento o dato necesario para medir, priorizar, auditar o cerrar fase.

Documento mínimo de faseDocumento suficiente para arrancar una fase, aunque no sea su versión completa.

Documento completoDocumento desarrollado con todos sus componentes, aplicable ordinariamente dentro de la fase que corresponde.

Capacidad habilitadaAcción que el sistema ya puede ejecutar porque tiene documentos, autoridad, datos, responsables y evidencia mínima.

Capacidad prohibida por cierre prematuroAcción que no puede activarse todavía porque faltan documentos habilitantes.

MIVSMinimum Institutional Viable System. Es el núcleo institucional mínimo viable. No reemplaza al sistema completo; permite iniciar Fase I sin romper trazabilidad, bienestar, gobernanza, datos ni auditoría.

CNVRMetodología territorial de Captura, Neutralización/Esterilización, Vacunación y Retorno. En el SMGIP no significa “capturar, operar y soltar”. Significa captura humanitaria, valoración clínica, identificación o prealta, esterilización cuando proceda, vacunación, registro RUAM, evaluación conductual y territorial, retorno georreferenciado o ruta alternativa, seguimiento y auditoría.

Retorno territorial responsableDecisión técnica documentada que permite devolver un animal comunitario a su territorio solo si cumple criterios de salud, conducta, bienestar, vacunación, esterilización cuando proceda, identificación, georreferencia, seguimiento y aceptación o viabilidad territorial.

PrealtaRegistro temporal justificado para animales aún no plenamente integrados. No equivale a alta plena. La alta plena requiere identificación primaria, validación sanitaria y expediente RUAM.

Alta plenaCondición de integración completa al SMGIP: microchip, RUAM, estatus sanitario, responsable o categoría operativa, evidencia y validación.

Estado documentalNivel de madurez de un documento dentro del árbol.

Release de faseCorte documental versionado que permite decir: “con estos documentos, esta fase puede iniciar o avanzar”.

V. ESTADOS DE MADUREZ DOCUMENTAL

E0 — No desbloqueadoEl documento todavía no debe escribirse. Solo puede existir ficha, propósito y dependencias.

E1 — Esqueleto permitidoPuede tener índice, alcance tentativo, preguntas pendientes y criterios de desbloqueo.

E2 — Borrador preliminarPuede redactarse parcialmente porque sus dependencias existen en borrador.

E3 — Borrador sustantivoPuede desarrollarse completo porque sus dependencias están en versión mínima estable.

E4 — Revisión técnica, jurídica, operativa o financieraPuede auditarse contra sus documentos habilitantes.

E5 — Versión activaPuede usarse para implementación, derivación, formatos, capacitación o fase.

IX. CIERRE CONCEPTUAL

Este árbol no describe papeles. Describe la manera en que una política pública compleja se vuelve escribible, implementable, auditable y defendible.

Su lógica es:

Documento maestro → fragmentación → núcleo rector → dependencias → fase mínima → operación controlada → auditoría → transición → regulación → exigibilidad plena.

La IA no debe usar este árbol como lista de temas. Debe usarlo como mapa de restricciones.

Git no debe usarse solo como almacenamiento. Debe usarse como sistema de memoria institucional, versionado, trazabilidad y control de cambios.

El SMGIP no se escribe de una vez. Se descompone desde su documento final, se reconstruye por dependencias y se activa por fases.
